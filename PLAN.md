# План развития проекта

Цель проекта — не расширять продукт, а встроить агента в цикл разработки: issue → triage → PR → ревью → доработки → ночные проверки.

## Баги (исправить)

### BUG-1: CI e2e-тесты не запускаются
Workflow `e2e.yml` выполняет `cd frontend && npx playwright test`, но тесты лежат в `tests/` в корне проекта. CI всегда падает или не находит тесты. Также Playwright-артефакты ищутся в `frontend/playwright-report/`, а реальный отчёт будет в `tests/playwright-report/`.

### BUG-2: SPA fallback отдаёт HTML на неизвестные API-роуты
Catch-all middleware в `backend/src/index.ts` (строка 37) возвращает `index.html` для *любого* 404. Если запросить несуществующий API-путь (`/api/unknown`), клиент получит HTML вместо JSON-ошибки. Нужно добавить проверку: для путей начинающихся с `/api` — возвращать JSON 404.

### BUG-3: SlotsDialog не сбрасывает startDate при смене типа события
`startDate` — внутренний стейт компонента. Когда диалог закрывается и открывается для другого eventType, компонент не размонтируется, старый `startDate` сохраняется. В результате `useApi` запрашивает слоты нового типа с фильтром по дате от предыдущего. Нужно сбрасывать `startDate` при изменении `eventType.id`.

### BUG-4: CLAUDE.md содержит неверный путь к e2e-тестам
В CLAUDE.md указано: «Playwright, директория `frontend/e2e/`». На самом деле тесты лежат в `tests/`. Это дезинформирует агента при работе с кодовой базой.

## Фичи (добавить)

### FEAT-1: Отмена бронирования (DELETE /api/bookings/:id)
Владелец может отменить бронь. Статус меняется на `cancelled`, слот освобождается (убирается из `reservedKeys`). На фронте — кнопка «Отменить» в списке бронирований. Требует расширения TypeSpec-контракта.

### FEAT-2: Удаление типа события (DELETE /api/event-types/:id)
Владелец может удалить тип события. Связанные бронирования помечаются как отменённые. На фронте — кнопка удаления в карточке. Требует расширения TypeSpec-контракта.

### FEAT-3: Ночной scheduled workflow с Lighthouse
GitHub Actions workflow по расписанию (каждую ночь), который запускает Lighthouse audit на деплое и сохраняет отчёт как артефакт. Демонстрация автоматических регулярных проверок агентом.

### FEAT-4: Клиентская валидация форм
Валидация в BookingDialog и CreateEventTypeDialog до отправки на сервер: мгновенная обратная связь, подсветка ошибок. Повторяет правила из `validation.ts` (длительность 15–480, email-формат, непустые поля).

## Порядок выполнения

1. **BUG-1** (CI e2e) — без работающего CI нельзя доверять результатам дальнейших PR
2. **BUG-4** (CLAUDE.md) — агент должен видеть корректную структуру проекта
3. **BUG-2** (SPA fallback) — мешает отладке и вносит путаницу в ответы API
4. **BUG-3** (SlotsDialog) — пользовательский баг, видимый на UI
5. **FEAT-1** (отмена бронирования) — ключевая недостающая фича, хороший кандидат на полный цикл issue → PR → ревью
6. **FEAT-2** (удаление типа события) — дополняет FEAT-1, аналогичный цикл
7. **FEAT-3** (Lighthouse scheduled) — организационная обвязка, демонстрация schedule workflow
8. **FEAT-4** (клиентская валидация) — улучшение UX, можно использовать для демонстрации доработок после ревью
